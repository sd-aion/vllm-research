# vLLM Attention Backend Legend Explained

The table in `vllm/docs/design/attention_backends.md` is not handwritten documentation. It is auto-generated by:

- `tools/pre_commit/generate_attention_backend_docs.py`

That matters because the table is mostly a rendered view of backend capability methods in code, not an independent spec.

The two core mechanisms behind the table are:

- Capability declaration on the backend class, mostly in `vllm/v1/attention/backend.py`
- Backend selection and validation in platform-specific selectors such as `vllm/platforms/cuda.py`

The main validation entry point is:

- `AttentionBackend.validate_configuration(...)` in `vllm/v1/attention/backend.py`

That function checks most of the legend columns directly.

## The Minimal Mental Model

When vLLM builds an attention layer, it does roughly this:

1. Build an `AttentionSelectorConfig` in `vllm/v1/attention/selector.py`
2. Ask the current platform for a backend class in `vllm/platforms/cuda.py`, `vllm/platforms/rocm.py`, or `vllm/platforms/cpu.py`
3. Validate candidate backends with `AttentionBackend.validate_configuration(...)`
4. Instantiate the chosen backend's metadata builder, runtime impl, and KV-cache shape/layout rules

The attention layer entry point is:

- `vllm/model_executor/layers/attention/attention.py`

That file is important because it is where model config gets turned into:

- `head_size`
- `num_heads`
- `num_kv_heads`
- `kv_cache_dtype`
- `has_sink`
- `use_mm_prefix`
- `attn_type`

Those values feed directly into backend selection.

## Where The Legend Comes From

These are the most important source files for understanding the legend:

- `vllm/v1/attention/backend.py` The abstract capability surface: dtype, KV dtype, block size, head size, sink, non-causal, sparse, MM prefix, attention type, compute capability.
- `vllm/v1/attention/selector.py` Builds the runtime selection config and passes it to the platform selector.
- `vllm/platforms/cuda.py` Chooses valid CUDA backends and raises errors if a selected backend is incompatible.
- `tools/pre_commit/generate_attention_backend_docs.py` Explains how the markdown table is derived from AST-parsed backend code.
- `vllm/model_executor/layers/attention/attention.py` Where layer-level settings become backend-selection inputs.
- `vllm/v1/kv_cache_interface.py` KV-cache spec definitions and memory accounting.
- `docs/serving/context_parallel_deployment.md` Best high-level explanation of DCP.

Representative backend files worth reading side by side:

- `vllm/v1/attention/backends/flash_attn.py`
- `vllm/v1/attention/backends/triton_attn.py`
- `vllm/v1/attention/backends/flex_attention.py`
- `vllm/v1/attention/backends/cpu_attn.py`

## Legend, Column By Column

## Dtypes

What it means:

- This is the set of model/attention compute dtypes a backend accepts for the Q/K/V path.
- In practice this is the dtype of the attention inputs and the attention math path, not a statement about every tensor in the entire model.

Where it comes from:

- `supported_dtypes` class variable in backend classes
- `supports_dtype(...)` in `vllm/v1/attention/backend.py`
- Checked by `validate_configuration(...)`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/v1/attention/backends/cpu_attn.py` Example: explicitly includes `torch.float32`, which is why the table shows `fp32` support for `CPU_ATTN`.
- `vllm/v1/attention/backends/flash_attn.py` Example: only `fp16` and `bf16`.

How to read the table correctly:

- If a backend lists `fp16, bf16`, it does not mean every feature of that backend works for every KV dtype and every hardware target. Other columns still constrain it.

## KV Dtypes

What it means:

- These are the supported storage formats for the KV cache, not the Q/K/V activation dtype.
- This is one of the most important columns for plugin work because many kernel integrations fail on KV-cache layout and quantization details, not on the attention math itself.

Where it comes from:

- `supported_kv_cache_dtypes` class variable
- Or a backend-specific override of `supports_kv_cache_dtype(...)`
- Parsed for docs by `parse_kv_cache_dtypes(...)` in `generate_attention_backend_docs.py`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/v1/kv_cache_interface.py` Defines KV quantization modes like FP8, per-token-head formats, NVFP4.
- `vllm/v1/attention/backends/flash_attn.py` Example: FP8 support is version- and hardware-gated.
- `vllm/v1/attention/backends/triton_attn.py` Example: supports more quantized KV formats including per-token-head quantization.

Practical interpretation:

- `auto` means “use the default non-quantized path chosen by config/platform”.
- `fp8`, `fp8_e4m3`, `fp8_e5m2`, `int8_per_token_head`, `nvfp4`, etc. each imply different KV-cache storage and kernel handling.

## Block Sizes

What it means:

- This is the set of KV-cache block sizes the backend kernel can work with.
- In vLLM, a “block” means a fixed-size chunk of tokens in the paged KV cache.

Where it comes from:

- `get_supported_kernel_block_sizes()`
- `supports_block_size(...)`
- `get_preferred_block_size(...)`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/v1/worker/utils.py` The worker negotiates common kernel block sizes across attention groups.
- `vllm/v1/kv_cache_interface.py` KV-cache specs and memory sizing depend on block size.
- `docs/design/paged_attention.md` Conceptual background for what a block is.

How to read `%N`:

- `%16` means “any framework block size that is a multiple of 16”.
- This is not just documentation shorthand. It matches real logic in `supports_block_size(...)`.

Important nuance:

- The framework-level block size and the kernel’s effective block size are not always identical.
- vLLM supports “virtual block splitting” in some cases, so a backend can accept a larger framework block size as long as it is compatible with the kernel’s minimum granularity.

What that means concretely:

- The KV-cache manager may allocate memory in larger blocks, for example `256` tokens per KV block.
- But the underlying attention kernel may only know how to operate on smaller blocks, for example `64`.
- In that case, vLLM does not force the whole system to allocate native `64`-token blocks. Instead, it keeps the larger allocation unit and logically splits each allocated KV block into multiple kernel-visible blocks.
- So one physical/allocation block of `256` tokens becomes four logical/kernel blocks of `64` tokens each.

Example files:

- `vllm/v1/attention/backends/flash_attn.py` Returns `MultipleOf(16)` in the normal case.
- `vllm/v1/attention/backends/flashinfer.py` Uses fixed sizes like `16, 32, 64`.
- `vllm/v1/attention/backends/cpu_attn.py` Also uses `%16`.

## Head Sizes

What it means:

- This is the supported per-head hidden dimension, not the full model hidden size.
- If the model has hidden size 4096 and 32 attention heads, the head size is 128.

Where it comes from:

- `get_supported_head_sizes()`
- `supports_head_size(...)`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/model_executor/layers/attention/attention.py` The layer passes `head_size` into selection.
- Model definitions such as: - `vllm/model_executor/models/gemma.py` - `vllm/model_executor/models/falcon.py` - `vllm/model_executor/models/deepseek_v2.py`

Nuance with TP:

- Tensor parallelism changes local `num_heads` and local `num_kv_heads`, but `head_size` itself usually stays the same.

Examples:

- `FLASH_ATTN` often shows `Any` because it overrides `supports_head_size(...)` more flexibly.
- `CPU_ATTN` enumerates specific allowed sizes.

Short note on `FLASH_ATTN`:

- The doc generator mainly looks for an explicit `get_supported_head_sizes()` list. `FLASH_ATTN` does not provide one, so the table renders `Any`.
- In reality, `FLASH_ATTN` narrows support in `supports_head_size(...)` inside `vllm/v1/attention/backends/flash_attn.py`, for example requiring head sizes divisible by 8 and then applying version-specific upper bounds.

## Sink

What it means:

- “Sink” means attention sink support, used for StreamingLLM-style behavior.
- In practice this is an extra per-head bias/tensor path that some kernels support and some do not.

Short note on what an attention sink is:

- An attention sink is a small learned or configured per-head bias term that gives some tokens a persistent baseline amount of attention mass, even when the normal causal/local attention pattern would otherwise make them easy to forget.
- In vLLM, this shows up as an extra `sinks` tensor passed into certain attention backends and kernels.

Where it comes from:

- `supports_sink(...)`
- Selection input `has_sink` in `vllm/v1/attention/selector.py`
- Layer-side detection in `vllm/model_executor/layers/attention/attention.py`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/model_executor/layers/attention/attention.py` Sets `self.has_sink = extra_impl_args.get("sinks") is not None`
- `vllm/v1/attention/backends/flash_attn.py`
- `vllm/v1/attention/backends/flashinfer.py`
- `vllm/v1/attention/backends/triton_attn.py`
- `vllm/v1/attention/ops/triton_unified_attention.py`

Model examples that pass sinks:

- `vllm/model_executor/models/gpt_oss.py`
- `vllm/model_executor/models/laguna.py`
- `vllm/model_executor/models/mimo_v2.py`

Important nuance:

- Sink support is often conditional.
- Example: `FlashAttention` may support sinks only for certain hardware/version combinations.
- Example: `FlashInfer` sink support is tied to TRTLLM attention on Blackwell.

## Non-Causal

What it means:

- This means bidirectional attention support inside the decoder-style paged-KV path.
- It does not mean the same thing as encoder attention.
- In plain terms: causal attention only lets token `i` attend to tokens `<= i`, while non-causal attention lets token `i` attend both left and right within the active attention region.

The key comment is in:

- `vllm/v1/attention/backend.py`

That file explicitly says non-causal attention is different from `ENCODER_ONLY`.

Where it comes from:

- `supports_non_causal(...)`
- `use_non_causal` in `AttentionSelectorConfig`
- Checked by `validate_configuration(...)`

Important code:

- `vllm/config/attention.py` Has `use_non_causal`
- `vllm/v1/attention/selector.py` Passes `use_non_causal=vllm_config.attention_config.use_non_causal`
- `vllm/v1/attention/backend.py` Rejects backends that do not implement `supports_non_causal(...)`
- `vllm/v1/attention/backends/flash_attn.py`
- `vllm/v1/attention/backends/flex_attention.py`
- `vllm/v1/spec_decode/dflash.py` Real user of non-causal selection

Practical interpretation:

- If a backend says `Non-Causal = ✅`, it can run a decoder-layer attention kernel without enforcing a causal mask.
- That is useful for some speculative or specialized decoding flows.

Where it is actually used:

- `DFlash` is the clearest direct usage in vLLM today. In `vllm/v1/spec_decode/dflash.py`, the draft-model config is rewritten with `use_non_causal=not self.dflash_causal`, so DFlash can intentionally request bidirectional attention for the draft pass.
- Some decoder-family models are also run in a bidirectional mode for embedding-style or special inference setups. You can see comments for this pattern in: - `vllm/model_executor/models/llama.py` - `vllm/model_executor/models/qwen2.py` - `vllm/model_executor/models/qwen3.py` These note that overriding `is_causal=False` enables bidirectional attention for certain embedding-oriented uses.

Why this is separate from encoder attention:

- Encoder attention is a separate attention type with different model semantics and metadata flow. Uses cross attention.
- Non-causal here means: keep the decoder-style paged-KV execution path, but remove the strict left-to-right mask.
- So a backend may support `ENCODER` attention, or non-causal decoder attention, or both. They are related but not interchangeable capabilities.

## MM Prefix

What it means:

- This is support for multimodal prefix full attention.
- The typical case is a model that needs some token ranges, often image/document prefix ranges, to behave bidirectionally or with special full-attention handling.

Where it comes from:

- `supports_mm_prefix(...)`
- `use_mm_prefix` in selector config
- Checked by `validate_configuration(...)`

Important code:

- `vllm/v1/attention/backend.py`
- `vllm/model_executor/layers/attention/attention.py` Sets `use_mm_prefix` from `model_config.is_mm_prefix_lm`
- `vllm/config/model.py` `is_mm_prefix_lm`
- `vllm/v1/worker/gpu_model_runner.py` Builds and attaches `mm_prefix_range` metadata
- `vllm/v1/attention/backends/triton_attn.py` Has explicit `mm_prefix_range` metadata handling
- `vllm/v1/attention/backends/flex_attention.py` Another concrete implementation
- `vllm/v1/attention/ops/triton_unified_attention.py` Consumes `mm_prefix_range`

Practical interpretation:

- This is not generic “multimodal support”.
- It is specifically about whether the backend can express and execute these prefix attention ranges correctly.

### MM Prefix Deep Dive

The shortest useful definition is:

- `mm_prefix` is a range-based bidirectional override on top of normal decoder attention.
- vLLM keeps the normal causal decoder mask by default, then selectively allows full bidirectional attention inside certain multimodal token spans.

Base causal decoder mask:

$$ M_{\text{base}}(i,j) = [j \le i] $$

Multimodal prefix override mask:

$$ M_{\text{mm}}(i,j) = \bigvee_r \left[i \in r \land j \in r\right] $$

Final effective mask: $$ M_{\text{final}} = M_{\text{base}} \lor M_{\text{mm}} $$

In Triton, the logic is a little more specific:

$$ M_{\text{final}} = (\text{causal} \land \text{sliding/chunk constraints}) \lor \text{mm\_prefix} $$

The important meaning is:

- outside the marked multimodal ranges, attention is still normal decoder causal attention
- inside a marked multimodal range, tokens may attend to each other bidirectionally
- the `mm_prefix` override is applied after the sliding-window restriction, so it can restore full connectivity inside those multimodal spans

What the ranges are:

- vLLM stores `mm_prefix` as one or more `(start, end)` token ranges per request
- each range marks a token span that should behave like a bidirectional multimodal prefix/document block

The metadata form is visible in:

- `vllm/v1/attention/backends/triton_attn.py`

Specifically:

- `mm_prefix_range: dict[int, list[tuple[int, int]]]`
- `mm_prefix_range_tensor: torch.Tensor | None`

How the data flows:

1. The model is identified as an mm-prefix LM through `model_config.is_mm_prefix_lm`
2. The GPU model runner walks the request’s multimodal features and extracts embedding ranges
3. Those ranges are collected into `req_doc_ranges`
4. The ranges are attached to each attention metadata object
5. Triton/FlexAttention backends consume those ranges when building the final attention mask

The key code path is:

- `vllm/config/model.py` - `is_mm_prefix_lm`
- `vllm/v1/worker/gpu_model_runner.py` - constructs `req_doc_ranges` - `_set_mm_prefix_range_for_metadata(...)`
- `vllm/v1/attention/backends/triton_attn.py` - packs ranges into `mm_prefix_range_tensor`
- `vllm/v1/attention/ops/triton_unified_attention.py` - receives the tensor
- `vllm/v1/attention/ops/triton_attention_helpers.py` - applies the override in the mask

Model-specific nuance:

- Not every model applies `mm_prefix` identically across all layers
- For example, Gemma4 explicitly clears `mm_prefix_range` for some full-attention layers and keeps it only where the model wants bidirectional multimodal behavior

See:

- `vllm/model_executor/models/gemma4_mm.py`

One more practical nuance:

- The runner may skip very large multimodal ranges when they exceed the sliding-window budget, rather than blindly applying bidirectional attention everywhere - if range length > sliding_window, it skips that range - otherwise it keeps it

That filtering happens in:

- `vllm/v1/worker/gpu_model_runner.py`

Best files to read for `mm_prefix` specifically:

1. `vllm/v1/worker/gpu_model_runner.py`
2. `vllm/v1/attention/backends/triton_attn.py`
3. `vllm/v1/attention/ops/triton_attention_helpers.py`
4. `vllm/v1/attention/ops/triton_unified_attention.py`
5. `vllm/v1/attention/backends/flex_attention.py`
6. `vllm/model_executor/models/gemma4_mm.py`

## DCP

What it means:

- DCP = Decode Context Parallelism.
- It shards decode-time KV cache across GPUs along the sequence-length dimension after TP has already sharded by KV heads.

Best high-level doc:

- `docs/serving/context_parallel_deployment.md`

The most important nuance:

- The docs table’s DCP column is not derived from a `supports_dcp()` method.
- In `generate_attention_backend_docs.py`, DCP support is inferred by checking whether the backend impl class has `can_return_lse_for_decode = True`.

Why LSE matters:

- DCP needs per-rank attention results to be combined correctly.
- That requires softmax log-sum-exp information during decode.

Where it is enforced:

- `vllm/v1/worker/cp_utils.py` Runtime assertion: DCP requires `layer_impl.need_to_return_lse_for_decode`
- `vllm/v1/attention/backend.py` `AttentionImplBase.__new__` sets `need_to_return_lse_for_decode` when `dcp_world_size > 1` and the impl can return LSE
- `vllm/config/model.py` Validates DCP constraints for GQA/MQA
- `vllm/v1/kv_cache_interface.py` DCP changes KV memory accounting; sliding-window spec currently rejects DCP
- `vllm/v1/worker/gpu_model_runner.py` Prepares DCP-local sequence lengths and block table behavior

Backend implementation examples:

- `vllm/v1/attention/backends/flash_attn.py` Has explicit DCP metadata, query all-gather, and LSE-based combine path
- `vllm/model_executor/layers/attention/mla_attention.py` MLA DCP path and additional caveats

## DCP Deep Dive

The most useful mental model is:

- tensor parallelism shards by heads
- DCP shards by sequence length / time

Very roughly, if a request has context length `T`, total KV heads `H_{kv}`, and head dimension `d`, then total KV storage scales like:

$$ \text{KV size} \propto T \cdot H_{kv} \cdot d $$

With plain tensor parallelism:

$$ \text{local KV} \propto \frac{T \cdot H_{kv} \cdot d}{tp} $$

But that ideal sharding only works while `tp <= H_{kv}`. Once `tp > H_{kv}`, some TP ranks have to replicate KV heads. DCP is the mechanism that reduces that replication by also sharding across the `T` dimension.

So with DCP, each rank stores only part of the decode context:

$$ \text{local context length} \approx \frac{T}{dcp} $$

The result is:

- TP first shards KV by head groups
- DCP then further shards KV by time blocks

Why DCP is nontrivial:

- each rank only has local KV shards `(K_r, V_r)`
- but softmax normalization should still be computed as if the query attended to the full context

Without DCP, one decode query uses:

$$ \text{Attn}(q, K, V) = \text{softmax}\left(\frac{qK^\top}{\sqrt{d}}\right)V $$

And it is why the docs generator uses decode-LSE capability as the proxy for the `DCP` column.

At a high level, a DCP decode step looks like:

1. each rank owns a local KV shard
2. query or query-related state may be gathered/shared across the DCP group
3. each rank computes local attention against its local KV
4. local outputs and LSE statistics are merged across ranks
5. the final corrected attention output is produced

The clearest concrete implementation is:

- `vllm/v1/attention/backends/flash_attn.py`

That file contains:

- DCP-specific metadata fields
- query gather across the DCP group
- local attention against local KV
- LSE-based merge using: - `cp_lse_ag_out_rs` - or `dcp_a2a_lse_reduce`

Important config constraints:

For GQA/MQA, DCP validation in `vllm/config/model.py` enforces:

1. `tp > total_num_kv_heads`
2. `dcp <= tp // total_num_kv_heads`
3. number of query heads per KV head must be divisible by `dcp`

So DCP is not an arbitrary extra parallelism knob. It is constrained by the model’s head geometry.

Why DCP does not increase GPU count:

- `decode_context_parallel_size` reuses GPUs already inside the TP group
- it does not launch extra workers

That is encoded in:

- `vllm/config/parallel.py`

Memory-accounting consequence:

- DCP changes KV memory sizing because each rank stores only a fraction of the full context
- `FullAttentionSpec.max_memory_usage_bytes(...)` scales effective max context length by the total CP world size (`dcp * pcp`)

That logic is in:

- `vllm/v1/kv_cache_interface.py`

One current restriction:

- sliding-window attention currently rejects DCP in `SlidingWindowSpec.max_memory_usage_bytes(...)`

See:

- `vllm/v1/kv_cache_interface.py`

Best files to read for DCP specifically:

1. `docs/serving/context_parallel_deployment.md`
2. `vllm/config/model.py`
3. `vllm/config/parallel.py`
4. `vllm/v1/worker/cp_utils.py`
5. `vllm/v1/worker/gpu/cp_utils.py`
6. `vllm/v1/worker/gpu_model_runner.py`

## Two Important “Meta” Files

If you want to understand why the table says what it says, read these before diving into every backend:

1. `vllm/v1/attention/backend.py` This is the real capability contract.
2. `tools/pre_commit/generate_attention_backend_docs.py` This is how those capabilities become the markdown table.

These two files explain almost the entire legend.

## How The Table Is Generated In Practice

The generator script does not import GPU libraries. It AST-parses backend code and extracts:

- `supported_dtypes`
- `supported_kv_cache_dtypes`
- `get_supported_kernel_block_sizes()`
- `get_supported_head_sizes()`
- `supports_attn_type(...)`
- `supports_sink(...)`
- `supports_non_causal(...)`
- `supports_mm_prefix(...)`
- `is_sparse(...)`
- `supports_compute_capability(...)`

For DCP, the generator uses a proxy:

- It finds the backend’s impl class from `get_impl_cls()`
- It walks the impl inheritance chain
- It checks whether `can_return_lse_for_decode` is set

That is an important implementation detail because it means:

- The DCP column is partly structural, not just a simple backend-class boolean.

## What Matters Most For Plugin Work

If you are adding a new attention backend or plugin, the highest-signal columns are:

- `KV Dtypes`
- `Block Sizes`
- `Head Sizes`
- `DCP`
- `Attention Types`

Those are the columns most likely to force real implementation work rather than a simple registration change.

The lowest-risk columns to start with are:

- `Dtypes`
- `Compute Cap.`

Those are still important, but they are usually easier to declare and validate than KV-cache format and DCP behavior.

- `Dtypes`, `KV Dtypes`, `Block Sizes`, `Head Sizes`, `Sink`, `Non-Causal`, `Sparse`, `MM Prefix`, `Attention Types`, and `Compute Cap.` mostly come from `AttentionBackend` capability methods.
- `DCP` is special because it depends on decode-time LSE support in the impl, plus runtime DCP checks in the worker/model-runner path.
- The markdown is generated from code, so the code is the source of truth.
